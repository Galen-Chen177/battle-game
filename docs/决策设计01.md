# Battle Engine 设计决策（一）：事件驱动架构（Event Driven）

> 更新时间：2026-07-14

## 背景

Battle Engine 的目标是成为一个**独立于任何前端技术的战斗计算核心**。

它不依赖：

* React
* Vue
* Unity
* Godot
* Flutter
* CLI

它只负责计算战斗。

因此，Battle Engine 与前端必须彻底解耦。

---

# 设计原则

Battle Engine 只负责一件事情：

> **根据当前战场状态，计算整场战斗。**

计算完成后，将整个战斗过程转换成一系列 Event。

Battle Engine 不负责：

* 人物移动
* 技能动画
* 飘字
* 音效
* 血条动画
* UI 刷新
* WebSocket 推送速度

这些全部属于前端职责。

---

# 后端职责

后端输入：

```text
BattleConfig
```

例如：

* 双方阵容
* 属性
* 技能
* Buff
* 宠物
* 战斗规则

Battle Engine：

```text
BattleConfig
        │
        ▼
   Battle Engine
        │
        ▼
 BattleResult
```

Battle Engine 一次性完成整场战斗计算。

计算过程中不会：

* Sleep
* Tick
* Timer
* 等待动画播放

整个过程通常只需要几毫秒即可完成。

---

# 前端职责

前端只负责播放。

例如收到：

```text
AttackEvent
```

播放攻击动画。

收到：

```text
HealEvent
```

播放回血动画。

收到：

```text
DeadEvent
```

播放死亡动画。

前端永远不会重新计算：

* 伤害
* 命中
* Buff
* 技能
* AI

所有结果均来自 Battle Engine。

---

# 为什么不使用 WebSocket 实时推送？

Battle Engine 是一个确定性的模拟器。

一场持续 120 秒的战斗：

实际计算时间可能只有：

```text
2ms
```

如果使用 WebSocket：

```text
Battle Engine

↓

Attack

↓

Sleep

↓

Attack

↓

Sleep

↓

Attack
```

实际上是在浪费服务器资源。

更合理的方式是：

```text
Battle Engine

↓

一次计算完成

↓

返回 BattleResult

↓

前端自行播放
```

因此：

Battle Engine 不负责播放节奏。

播放速度完全交给前端。

例如：

* 1 倍速
* 2 倍速
* 4 倍速
* 跳过动画
* 回放

都无需重新计算。

---

# Event 是唯一输出

Battle Engine 永远只输出 Event。

未来不会直接输出：

```text
张三攻击哥布林
```

而是：

```text
AttackEvent
```

所有表现均由 Event 驱动。

整个项目采用 Event Driven Architecture。

---

# Event 设计

目前 Event 为一个结构体。

后续计划升级为接口。

例如：

```go
type Event interface {
    Time() float64
    Type() string
}
```

所有事件均实现该接口。

例如：

```text
AttackEvent
HealEvent
DeadEvent
BuffAddEvent
BuffRemoveEvent
SummonEvent
BattleStartEvent
BattleEndEvent
```

Battle Engine 输出：

```go
[]Event
```

前端根据 Type() 决定如何播放。

---

# BattleResult

Battle Engine 最终返回 BattleResult。

建议结构：

```go
type BattleResult struct {
    Winner TeamID

    Duration float64

    Events []Event
}
```

BattleResult 是 Battle Engine 唯一输出对象。

---

# 为什么采用 Event 接口？

采用接口而不是一个巨大的 Event 结构体，有以下优势：

## 可扩展

以后增加事件类型时，无需修改已有字段。

例如：

新增：

```text
BuffAddEvent
```

不会影响：

```text
AttackEvent
```

---

## 类型安全

每种 Event 拥有自己的数据结构。

例如：

AttackEvent：

```text
Attacker
Target
Damage
Critical
```

HealEvent：

```text
Caster
Target
Heal
```

不会出现大量无意义字段。

---

## 更符合 Go 风格

Go 更推荐：

多个小类型

而不是：

一个拥有几十个字段的大结构体。

---

# 战斗超时

Battle Engine 必须具备超时保护。

原因：

理论上存在无法结束的战斗。

例如：

```text
A 每秒回复 100 HP

B 每秒造成 80 伤害
```

双方永远不会死亡。

如果没有保护：

```text
Run()

↓

无限循环
```

因此 Battle Engine 必须支持超时终止。

建议增加：

```go
type BattleConfig struct {
    MaxBattleTime float64
}
```

例如：

```text
300 秒
```

当模拟时间超过：

```text
MaxBattleTime
```

立即结束战斗。

返回：

```text
BattleTimeoutEvent
```

并在 BattleResult 中标记：

```text
Timeout = true
```

最终结果：

```go
type BattleResult struct {
    Winner TeamID

    Timeout bool

    Duration float64

    Events []Event
}
```

BattleTimeout 并不是异常。

它属于 Battle Engine 的一种正常结束状态。

---

# 总结

Battle Engine 的职责只有四项：

1. 根据配置计算整场战斗。
2. 生成完整 Event 流。
3. 返回 BattleResult。
4. 提供稳定、可复用的战斗计算能力。

除此之外的任何事情（动画、UI、播放速度、音效、网络同步等）均不属于 Battle Engine 的职责。

坚持这一原则，可以保证 Battle Engine 能够长期演进，并被不同的客户端复用。
